If you're new to Bluprynt, these terms recur in every doc. Bookmark this page.

## CCID (Cross-Chain Identity)

A **CCID** is a Bluprynt-native, content-addressed identifier for any participant in the system — an issuer, an asset, a regulator. A CCID is:

- **Content-addressed** — derived from the identity's canonical data, so the same entity produces the same CCID everywhere.
- **Cross-chain** — one CCID can carry wallets on Ethereum, Solana, Stellar, Aptos, and XRPL simultaneously.
- **Registry-published** — each CCID is published to on-chain identity-registry contracts so downstream verifiers can resolve it without calling Bluprynt.

## Identity

An **identity** is the live, mutable record of a CCID plus its wallets, accounts, and registry deployments. Types: `asset`, `issuer`, `regulator`.

## Attestation

An **attestation** is a signed statement about an identity. Attestations live on-chain (via EAS, SAS, Stellar AS, Aptos, or XRPL) and reference large payloads via IPFS. Each attestation has:

- A **type** — `kyi`, `mica`, `regd`, `proof-of-collateral`, or custom.
- A **schema** — defines the fields and validation rules for its type.
- A **status** — `initialized` → `uploaded_to_ipfs` → `pending_approval` → `published` (or `revoked` / `rejected` / `expired`).

## Smart Disclosure

A **Smart Disclosure** is a structured regulatory filing generated from a jurisdiction-specific form. A disclosure is authored once in the Admin Console, rendered as a signed PDF, uploaded to IPFS, and committed to the chain via a disclosure attestation.

## Credential

A **credential** is an attestation of a specific type that proves the holder meets a particular requirement — e.g., *"this asset holds a US Reg D exemption"* or *"this issuer has a MiCA license from AMF."* Two kinds: `license` and `disclosure`.

## Policy module

A **policy module** is an on-chain program that reads an identity's credentials and decides whether a token operation is permitted. Policy modules are referenced by tokens at mint / transfer / burn time.

Read more in the [Policy Engine white paper](https://github.com/blupryntco/policy-engine).

## CAIP-2 / CAIP-10

Bluprynt uses the **CAIP-2** (chain) and **CAIP-10** (account) standards to address chains and accounts uniformly. Every attestation API endpoint is keyed on `{namespace}:{reference}:{address}`.

## Attestation type

A family of attestations sharing a schema and a payload transformer. Bluprynt ships four out-of-box: **KYI**, **MiCA**, **RegD**, **Proof-of-Collateral**. New types are added by implementing a handler in `bluprynt-attestation`.

## Identity registry

An on-chain contract that maps CCIDs to their wallets and metadata. Bluprynt's registry contracts are Chainlink-ACE-compatible.

## Supervised asset

An asset whose issuer has signed up for Bluprynt, completed KYB, and filed at least one disclosure. Supervised assets show up in Compliance Hub for regulators and counterparties.
