Bluprynt is a stack of cooperating services. Each product surface sits on top of the same core primitives — identities, attestations, and policy modules — so data stays consistent across products and across chains.

## The four layers

<Steps>
  <Step title="Identity">
    A **CCID** (Cross-Chain Identity) anchors every participant — issuer, asset, regulator. Identities carry wallets and accounts across chains and publish to on-chain identity-registry contracts.

    Learn more: [CCID](/architecture/ccid), [KYI](/core-products/kyi/introduction).
  </Step>
  <Step title="Disclosures">
    **Smart Disclosures** turn regulated filing forms into structured documents. Bluprynt generates a PDF, writes the artifact to IPFS, and writes a cryptographic attestation on-chain so downstream verifiers can confirm authenticity.

    Learn more: [Smart Disclosures](/core-products/smart-disclosures/introduction), [Attestation schemas](/architecture/attestation-schemas).
  </Step>
  <Step title="Supervision">
    **Compliance Hub** surfaces the state of every supervised asset — disclosures filed, attestations written, policy modules attached, credentials active. Regulators and counterparties get a single pane of glass.

    Learn more: [Compliance Hub](/core-products/compliance-hub/introduction).
  </Step>
  <Step title="Credentials">
    **Credentials** are type-specific attestations issued to identities: licenses, disclosures, jurisdictional eligibility. Policy modules reference credentials at token-operation time.

    Learn more: [Policy Engine](/architecture/policy-engine).
  </Step>
</Steps>

## How pieces interact

<AccordionGroup>
  <Accordion icon="file-plus" title="Publishing a disclosure">
    1. User submits a form in the Admin Console.
    2. `bluprynt-api` validates against the JSON schema and generates the artifact.
    3. Rails posts to `bluprynt-attestation`, which uploads the artifact to IPFS and writes the on-chain attestation via the appropriate chain adapter.
    4. The attestation's status transitions: `uploaded_to_ipfs` → `pending_approval` → `published`.
    5. `bluprynt-attestation` calls the Rails-side webhook `POST /web3/sas/kyi_webhook` to notify downstream consumers.
  </Accordion>
  <Accordion icon="id-card" title="A KYI lookup">
    1. Partner embeds the KYI widget via `@bluprynt/kyi-widget-sdk` with a short-lived JWT.
    2. Widget calls into `bluprynt-fe`'s `/widget/kyi` route (CSP-gated via `ALLOWED_EMBED_ORIGINS`).
    3. Request routes to `bluprynt-kyi`, which resolves the CCID and surfaces verified issuer details + attestations.
  </Accordion>
  <Accordion icon="gears" title="A policy check">
    1. At token-operation time (mint / transfer / burn), an on-chain policy module reads the issuer's attestations and active credentials.
    2. If requirements aren't met, the transaction reverts.
    3. If met, it executes. The attestation hub may write a new attestation capturing the event.
  </Accordion>
</AccordionGroup>
