Every Smart Disclosure moves through six states. This page walks each one.

## The state machine

```
 initialized → drafting → validating → submitted → publishing → published
                   │                                   │
                   └── amended ──┐                     └─► attested on-chain
                                 ▼
                            new revision
```

## Phases in detail

<Steps>
  <Step title="Initialize">
    You pick **jurisdiction + regulation + asset type + document type** in the Admin Console. Bluprynt creates an empty document record, sets its status to `initialized`, and loads the right JSON schema for the form.
  </Step>
  <Step title="Draft">
    You fill in fields (with optional AI-assisted autofill). Draft state is saved continuously. You can exit and return.
  </Step>
  <Step title="Validate">
    Clicking **Validate** runs every field through the schema's validation rules (type, format, required, conditional visibility, inter-field consistency).

    <Tip>
      Validation errors link directly to the offending field. No more "field 12 invalid" without context.
    </Tip>
  </Step>
  <Step title="Submit">
    Once validation passes, click **Submit**. The status transitions to `submitted`. At this point the document is **immutable** — further changes require filing an amendment.
  </Step>
  <Step title="Publish to chain">
    Click **Publish**. Pick target chain(s). Bluprynt:
    1. Generates a signed PDF via Prawn + wkhtmltopdf.
    2. Uploads the PDF to IPFS — returns a CID.
    3. Calls `POST /v1/attestations/{caip2}/{reference}/{type}` on `bluprynt-attestation` per chain.
    4. Each attestation goes `uploaded_to_ipfs` → `pending_approval` → `published` as it finalizes on-chain.
  </Step>
  <Step title="Verify">
    Click **View on-chain data** to see:
    - Transaction hash + explorer link per chain
    - IPFS CID
    - Attestation status

    From here, your disclosure is resolvable by anyone via the KYI widget, the attestation service's `GET /v1/attestations`, or native chain tooling.
  </Step>
</Steps>

## Amendments

To amend a published disclosure:

1. Open the document. Click **Amend**.
2. This creates a **new revision** — a sibling document with `parent_document_id` pointing at the original.
3. Edit, validate, submit, publish. The new revision publishes a new on-chain attestation that references the original.
4. Both the original and the amendment remain resolvable — the history is preserved, not overwritten.

## Revocation

If a disclosure needs to be withdrawn:

1. Open the document. Click **Unpublish**.
2. Bluprynt writes a revocation attestation on each chain the document was published to.
3. The attestation's status transitions to `revoked`.
4. KYI lookups now show `revoked` alongside the original attestation.

<Note>
  **Attestations are append-only on-chain.** "Unpublish" writes a new revocation attestation — it does NOT remove the original on-chain record.
</Note>

## States reference

| State | Meaning | Can edit? | On-chain? |
|---|---|---|---|
| `initialized` | Form selected, empty | Yes | No |
| `drafting` | In progress | Yes | No |
| `validating` | Running validation | No | No |
| `submitted` | Validated + locked | No (amend) | No |
| `publishing` | Chain writes in flight | No | Pending |
| `published` | Attestation confirmed | No (amend) | Yes |
| `revoked` | Withdrawn via revocation attestation | No | Yes |
| `amended` | Superseded by a revision | No | Yes (original kept) |
