## Quotas by plan

| Plan | Per minute | Per day | Burst |
|---|---|---|---|
| Starter | 60 | 10,000 | 10 rps |
| Growth | 300 | 100,000 | 30 rps |
| Scale | 1,500 | 1,000,000 | 100 rps |
| Enterprise | Custom | Custom | Custom |

Quotas are per **API key**. To raise limits, email [partners@bluprynt.com](mailto:partners@bluprynt.com).

## Response headers

Every response includes:

```
X-RateLimit-Limit: 300
X-RateLimit-Remaining: 287
X-RateLimit-Reset: 1714155000
```

- **`X-RateLimit-Limit`** — your current window's quota
- **`X-RateLimit-Remaining`** — requests left
- **`X-RateLimit-Reset`** — Unix timestamp when the window resets

## When you're rate-limited

A 429 response includes `Retry-After` in seconds:

```
HTTP/1.1 429 Too Many Requests
Retry-After: 12
Content-Type: application/json

{
  "error": "Rate limit exceeded",
  "code": "rate_limited",
  "details": { "retryAfterSeconds": 12 }
}
```

## Backoff guidance

- **On 429:** wait `Retry-After` seconds, then retry.
- **On 503:** exponential backoff — `min(60, 2^attempt)` seconds, max 5 attempts.
- **On 500:** same exponential backoff as 503.
- **On 4xx (not 429):** do not retry — fix the request.

## Batch-friendly endpoints

Some endpoints accept batches to reduce your request rate:

- `POST /v1/kyi/batch` — up to 100 CCIDs / wallets per call, counted as **one** request against your quota.
- `POST /v1/attestations/batch` — same pattern.

Using batches is the primary way to stay within limits during bulk ETL.

## Caching

Bluprynt returns `ETag` headers. Send `If-None-Match` on subsequent reads:

```
GET /v1/identities/{ccid}
If-None-Match: "abc123"
```

A 304 response uses **no** quota.
